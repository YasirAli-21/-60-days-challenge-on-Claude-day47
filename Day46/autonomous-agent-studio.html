📚 How the Stop-Check Governs the Loop

Each round the Evaluator scores the current draft against the quality rubric from a completely fresh API call — it never reuses a prior score. The Critic then reads that evaluation plus the draft and produces a targeted critique. The Improver receives both the evaluation and the critique (not just the raw draft) and rewrites the code accordingly. After the Improver responds, control returns to the Evaluator with the new draft — closing the loop — and the stop-check runs again, in this fixed priority order:

1. Plateau — if score improvement is below the configured delta for two consecutive rounds, the loop halts (diminishing returns).
2. Threshold — if the score crosses the target quality bar, the loop halts (goal achieved).
3. Hard cap — a safety fallback that halts the loop if neither condition fires within the configured max rounds, preventing runaway API spend.

Whichever condition fires first is logged and shown in the stop banner and final summary. Once the loop halts, the branch to the Final Reviewer activates — a distinct agent, outside the loop, that reviews the entire history (not just the last round) and writes the final verdict.

🏗️ Architecture Overview

7 agents, 2 phases. Setup phase (run once): Planner → Executor produces draft v0. Refinement loop (runs N times, N unknown until it stops): Evaluator → Critic → Memory Manager → stop-check → (Improver → back to Evaluator) or (→ Final Reviewer). Every arrow in the diagram above is a live fetch call to the Claude Messages API with the prior round's real text output threaded into the next prompt — there is no hardcoded round count and no simulated scoring.

🚀 Extension Ideas & Further Prompts

• Add a Safety Monitor agent that runs in parallel with the Evaluator to flag unsafe file-system operations before the Improver acts on them.
• Add a Test Generator agent that writes unit tests once the threshold is crossed, and loop those tests back through the Evaluator as an additional rubric input.
• Persist the memory log to localStorage-free server storage so multiple runs on related specs build a shared pattern library.
• Swap the Evaluator's rubric at runtime (correctness-only vs. quality-only vs. balanced) and compare final scores across pipelines.
• Try prompts like: "Add a Test Generator agent that validates the improver's code against edge cases before the next evaluation round," or "Make the plateau window configurable (3+ rounds) and show a trend sparkline."
